# 提示词书写宝典
本宝典基于20+实际业务提示词的优缺点分析，结合行业通用CRISPE提示词框架、大模型提示词工程最佳实践整理而成，覆盖90%以上大模型应用场景，可直接复用对应模块快速写出高匹配度的提示词。

---

## 一、提示词核心设计框架：CRISPE五要素
所有高质量提示词都可以拆解为以下5个核心模块，缺一不可：
| 模块 | 全称                            | 核心要求                                                     | 高频踩坑点                                                   |
| ---- | ------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| C    | 能力与角色（Capacity and Role） | 明确AI的专属身份、服务边界、核心能力，越精准匹配任务越好     | 角色定位泛化（如仅写「多功能助理」未说明具体服务范围）、角色和任务不匹配、服务边界模糊 |
| R    | 背景与洞察（Context/Insight）   | 明确任务的适用场景、面向受众、输入素材范围、参考依据优先级   | 缺少输入格式规范、参考范围不明确、场景/受众模糊导致输出不符合实际使用需求 |
| I    | 指令与任务（Instruction）       | 明确具体要完成的任务、执行逻辑、优先级规则、工具/能力调用规则 | 任务描述模糊、执行规则缺失、工具调用触发条件/参数不明确、多工具/多规则优先级未说明 |
| S    | 输出规范（Specification）       | 明确输出的格式、内容维度、粒度要求、风格要求、字数限制等     | 输出要求笼统（如仅写「输出结果」未说明格式/内容）、核心字段缺失、风格无明确标准 |
| E    | 约束与示例（Example/Exception） | 明确任务的约束条件、异常场景处理规则、正面/负面参考示例      | 无异常处理规则、无准确性约束、无示例导致执行偏差大           |

## 二、各模块详细书写规则
### 1. 【C：角色】书写要点
✅ 正确示例：`你是专门服务于大模型开发工程师岗位的简历项目经历审核专家，仅负责审核简历中项目经历板块的内容，其他板块内容不属于你的服务范围`
❌ 错误示例：`你是简历助理`
- 角色命名尽量垂直精准，不要使用「通用助理」「小助手」这类泛化表述
- 明确说明角色的服务边界，哪些需求属于/不属于服务范围
- 可补充角色的专业背景，增强输出的专业性（如「拥有5年互联网HR招聘经验」）

### 2. 【R：背景】书写要点
✅ 正确示例：`本次审核针对社招中级大模型应用开发岗求职简历，参考依据优先级为：预设审核规则>目标岗位JD>简历其他板块内容，仅审核项目经历板块内容`
❌ 错误示例：`帮我审核下简历`
- 明确任务的适用场景（如校招/社招、商务/日常、公域发布/内部使用）
- 明确输入素材的提交规范（如用户需标注「信息1：需求」「信息2：待处理文件」）
- 明确参考依据的优先级，避免出现信息冲突时输出混乱

### 3. 【I：指令】书写要点
✅ 正确示例：`优先调用三体知识库查询用户问题，若无匹配结果再调用通用联网搜索，调用搜索前需校验用户是否提供了明确的查询关键词，参数不全时先引导用户补充`
❌ 错误示例：`你可以调用搜索和知识库`
- 复杂任务要拆解为明确的执行步骤，每一步的要求清晰可落地
- 涉及工具调用需明确：
  - 每个工具的触发场景、必填入参
  - 工具调用优先级（如专用知识库>通用搜索）
  - 多工具联动规则（如做旅行规划时可同步调用天气、火车票工具整合信息）
  - 工具调用失败的重试规则（如最多重试1次，仍失败则告知用户）
- 明确无需调用工具的场景（如简单常识、基础计算类问题可直接回复）

### 4. 【S：输出规范】书写要点
✅ 正确示例：`输出为标准JSON格式，仅包含skill（个人技能数组）和project_exp（项目经历数组，每个元素包含项目名称、起止时间、担任角色、核心职责、项目成果5个字段），缺失字段标注“无”，无额外解释内容`
❌ 错误示例：`提取简历里的技能和项目经历`
- 明确输出的格式：如JSON、Markdown、表格、纯文本等，结构化格式需给出明确的字段/键名规则
- 明确输出的内容维度：如火车票汇总需包含车号、出发时刻、到达时刻、票价、票务状态等字段
- 明确输出的风格/粒度：如「300字以内精简总结」「小红书种草风格，搭配emoji，末尾带3-5个话题标签」「商务正式风格，避免口语化表述」
- 明确额外要求：如是否需要标注信息来源、是否需要高亮修改部分、是否需要保留原文件结构等

### 5. 【E：约束与示例】书写要点
✅ 正确示例：`所有抽取内容100%来自原文，禁止编造；若工具查询不到天气信息，需如实告知用户“暂时无法获取当前城市的天气数据，请稍后再试”；若用户需求超出旅游规划范畴，礼貌说明“我是专属旅游规划助理，目前仅能提供旅游相关服务哦”`
❌ 错误示例：`遵循规则处理`
- 必须包含**准确性约束**：所有内容必须基于给定素材/工具返回结果，禁止编造未提及的信息，涉及数据、专有名词需保留原表述
- 必须覆盖**核心异常场景处理规则**：
  - 输入信息不足时：主动引导用户补充必要参数，不要猜测执行
  - 处理/查询无结果时：如实告知用户，不得编造内容
  - 需求超出服务范围时：礼貌说明服务边界，引导用户提出合规需求
  - 内容违规/敏感时：直接拒绝处理并说明原因
- 复杂任务需附1-2组输入输出示例，具象执行标准，减少理解偏差

## 三、常见场景提示词模板
### 1. 信息抽取类（人名/字段/主体提取、内容结构化）
```
【角色】你是XX领域专属信息抽取专家
【背景】待抽取内容为XX场景下的文本，抽取字段规则如下：
- 字段1（变量名：xxx）：[抽取规则，如工号为8位纯数字、绩效等级取值为S/A/B/C/D]
- 字段2（变量名：xxx）：[抽取规则]
【指令】从用户提供的文本中抽取上述指定字段，重复内容自动去重，同字段多匹配时默认返回第一个符合规则的结果
【输出规范】输出为标准JSON格式，抽取不到的字段赋值为null，无任何额外解释内容
【约束】所有抽取内容100%来自原文，禁止编造；若文本无任何可抽取信息，返回所有字段为null的JSON
```

### 2. 内容生成类（文案/报告/规划/试题生成）
```
【角色】你是XX场景专属内容生成专家
【背景】本次生成内容适用于[场景，如小红书种草/朋友圈发布/企业内部报告]，面向[受众，如学生党/职场人/消费者]，风格要求为[如活泼亲民/商务正式/严谨专业]，可参考素材为[如搜索到的近3个月高赞笔记/用户提供的项目描述/企业面试真题]
【指令】完成[具体任务，如生成10道大模型工程师面试题/生成5天云南旅游行程]，执行逻辑：步骤1：调用XX工具查询所需实时信息；步骤2：整合信息生成内容；步骤3：校验内容符合风格要求
【输出规范】输出结构为[如XX模块/XX结构]，字数要求XX，格式为[如Markdown/带emoji的小红书格式/表格]
【约束】禁止编造信息，工具查询不到的内容标注“信息暂缺”；若素材不足需先告知用户，不得自行生成无依据内容
```

### 3. 审核校验类（简历审核/内容审核/信息核验）
```
【角色】你是XX领域专属审核专家
【背景】审核适用于[场景，如社招中级大模型岗求职简历审核/内容合规审核]，参考依据优先级为：预设审核规则>上下文>其他参考资料，审核维度如下：
- 维度1：[审核标准，如真实性：排查学历是否为国家认可的正规院校]
- 维度2：[审核标准，如规范性：技能表述需使用行业通用术语，剔除口语化表述]
【指令】对用户提供的待审核内容按照上述规则进行审核，找出所有待修改的问题
【输出规范】输出结构为：
【审核结论】：合格/不合格
【待修改问题列表】：每条包含问题描述、修改建议、严重等级（必填修改/建议优化）
【优化补充建议】：可选内容
【约束】所有审核结论必须有明确规则/上下文依据，禁止主观判断；若信息不足需引导用户补充对应内容
```

### 4. 工具调用类（天气/搜索/票务/旅行规划）
```
【角色】你是XX专属助理，可调用以下工具：
1. 工具1（名称）：功能[如查询指定城市指定日期的天气]，触发场景[用户查询天气相关内容时]，必填入参[城市、日期]
2. 工具2（名称）：功能[如联网搜索所有旅游相关实时信息]，触发场景[所有时效性强、非通用常识的旅游信息查询时]，必填入参[查询关键词]
【背景】服务范围为旅游规划相关需求，工具调用优先级：专用工具>通用工具，工具调用失败可重试1次，仍失败则告知用户
【指令】处理用户需求时优先匹配对应工具，调用前校验参数完整性，参数不全时主动引导用户补充
【输出规范】工具返回内容整理为通顺易懂的自然语言呈现，禁止暴露内部工具编码，可标注信息来源
【约束】所有时效性强的信息必须调用工具核实，禁止编造；若工具查询不到结果如实告知用户；需求超出旅游范围时礼貌说明服务边界
```

### 5. 多智能体路由类
```
【角色】你是多智能体路由决策节点，仅负责路由用户需求到对应智能体，不直接回答问题
【背景】可路由智能体及适用场景：
1. 智能体1（名称）：适用场景[用户有翻译诉求时]，调用需传递参数[待翻译内容]
2. 智能体2（名称）：适用场景[用户涉及员工信息增删改查时]，调用需传递参数[操作类型、员工工号、操作内容]
【指令】用户需求符合对应特征时调用对应智能体，多需求叠加时按[优先级规则，如翻译>员工信息处理>旅行规划]排序
【输出规范】用固定格式输出：<agent=智能体名称>传递参数</agent>，无额外内容
【约束】需求不在覆盖范围内时，回复固定话术：「当前需求暂不支持，请提出翻译、员工信息处理、旅行规划范围内的需求」
```

## 四、高频避坑指南
1. 禁止使用模糊表述：不要用「尽量」「大概」「好看」「体面」这类无明确判定标准的词汇，所有要求必须可落地、可量化
2. 禁止「规则留白」：不要仅说「遵循规则处理」却不给出具体的规则内容，所有判定标准必须明确列出
3. 禁止遗漏异常处理：至少覆盖「信息不足、无结果、需求超范围」三类常见异常的应对方案
4. 禁止角色泛化：尽量不要使用「通用助理」「多功能助手」这类模糊角色，角色越垂直，输出越精准
5. 禁止工具规则模糊：涉及工具调用必须明确触发场景、必填入参、优先级，不要仅说「可调用XX工具」
6. 禁止场景缺失：所有任务要明确适用场景、面向受众，不同场景的输出标准差异极大
